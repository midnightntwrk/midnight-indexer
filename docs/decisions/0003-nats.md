# Replacing Redis with NATS

## Context and Problem Statement

Currently – in the original Scala version and the so far implemented Rust rewrite – the Indexer makes use of [Redis](https://redis.io/) for inter-component communication, e.g. between the Chain Indexer and the Wallet Indexer. In the Rust rewrite we not only use publish-subscribe messaging, but also Redis set operations as an important feature.

While Redis is a well established product, its most popular [Rust client library](https://docs.rs/redis/latest/redis/) which we chose to use (see [0001-redis-pubsub-rust.md](./0001-redis-pubsub-rust.md)), has some drawbacks, which we have to workaround:
- Publish-subscribe is not properly supported for async Rust
- Set operations do not offer stream based watching, so we have to poll
- Reconnect logic is not conveniently available across the needed features

Therefore we are looking for a potential alternative for Redis.

## Decision Drivers

* Reduce workarouds in our codebase to reduce risk and maintenance burden
* Excellent support for async Rust
* Simple deployment and operations
* Excellent support for volatile (non-persistent) operations
* Match for our use cases:
    * Streaming publish-subscribe
    * Set operations
    * Streaming watching of set changes (not available in Redis)
* If available, make use of additional features to further simplify use cases not yet related to Redis

## Considered Options

* [NATS](https://nats.io/)
* [Kafka](https://kafka.apache.org/)
* [RabbitMQ](https://www.rabbitmq.com/)
* Keep Redis

## Decision Outcome

Kafka offers messaging and Kafka Streams makes key-value stores available to stream processors. But Kafka is a rather heavyweight system with its main focus on persistent message logs, which is not exactly what we need. Set operations are only available when running stream processing pipelines which is not at all what we need. Kafka is also know for being challenging to operate.

RabbitMQ is a populare messaging platform, but it does not offer any set operations.

So Kafka and RabbitMQ are not a good fit for the needs of the Indexer and therefore are no alternatives for replacing Redis.

The [NATS client library](https://docs.rs/async-nats/latest/async_nats/) has way better support for async Rust and for reconnections that the Redis client library. NATS allows for watching entries in its Key-Value feature, i.e. getting a stream of changes; no need for polling. It also has some advanced features like TTL we could utilize e.g. for storing active wallets/sessions; currently we have to "manually" periodically clean them in the DB. So overall we could significatly reduce our own code, in particular get rid of some non-functional (boilderplate) code.
